# Tag Management — Behavior Specification

## Overview

Tags are LLM-generated topic labels attached to knowledge items. They go through a user-controlled approval lifecycle that shapes future LLM behavior: approved tags are reused, rejected tags are suppressed.

---

## 1. Tag Lifecycle

```
[LLM generates tag] → pending → approved
                              → rejected → (item re-tagged without this tag)
```

### 1.1 States

| State | Meaning |
|---|---|
| `pending` | Generated by LLM, not yet reviewed by user |
| `approved` | User accepted this tag; LLM should reuse it |
| `rejected` | User rejected this tag; LLM must never generate it again |

### 1.2 Transitions

- **pending → approved:** User clicks approve (✓). Tag status set to `approved`. No re-tagging triggered.
- **pending → rejected:** User clicks reject (✗). Tag status set to `rejected`, `rejected_at` timestamp set. If an `itemId` is provided, that item is asynchronously re-tagged.
- **approved → rejected:** Possible via direct `POST /tags/reject` call. `rejected_at` is set.
- **rejected → approved:** Possible via `POST /tags/approve`. `rejected_at` is NOT cleared (kept for audit).
- **approved → pending:** Never happens. `upsertTag` with `status = pending` on an already-approved tag is silently ignored (the `ON CONFLICT` rule keeps `approved` status).

---

## 2. Tag Creation

**Trigger:** After a successful summarization, `processItem` calls `upsertTag(tag, 'pending')` for every tag the LLM returned.

**`upsertTag` conflict rule:**
```sql
ON CONFLICT(name) DO UPDATE SET
  status = CASE
    WHEN tags.status = 'approved' AND excluded.status = 'pending'
    THEN 'approved'   -- never downgrade approved tags
    ELSE excluded.status
  END,
  rejected_at = COALESCE(excluded.rejected_at, tags.rejected_at)
```

**Acceptance criteria:**
- New tag → inserted with `status = 'pending'`, `created_at = now`.
- Existing `approved` tag registered as `pending` → remains `approved` (no downgrade).
- Existing `rejected` tag re-registered → status updated to the new value; `rejected_at` preserved if not being set again.
- `rejected_at` only set when `status = 'rejected'`.

---

## 3. Re-tagging on Rejection

**Trigger:** `POST /tags/reject` with a non-empty `itemId`.

**Behavior (`retag` function in `src/server.ts`):**
1. Load item by `itemId`.
2. If item not found or has no transcript, return immediately (no-op).
3. Call `suggestTags(transcript, approvedTags, rejectedTags)` — the newly rejected tag is already in `rejectedTags` by this point.
4. If no tags returned, return immediately (no-op).
5. For each new tag, call `upsertTag(tag, 'pending')`.
6. Overwrite item's `tags` field with the new tag list.

**Acceptance criteria:**
- Re-tagging is fire-and-forget (errors silently discarded via `.catch(() => {})`).
- The rejected tag will not appear in the new tag list (it is in `rejectedTags` for the LLM).
- Approved tags are still candidates for re-use.
- Item's tag list is fully replaced (not merged) with new suggestions.
- If `suggestTags` returns empty, item retains its previous tags (no update called).

---

## 4. LLM Tag Guidance

Both `summarize` and `suggestTags` inject tag guidance into prompts:

**Approved tags prompt fragment:**
> Preferred existing tags (reuse these when they fit — ordered general to specific): `<comma-separated list>`

**Rejected tags prompt fragment:**
> Never use these tags (rejected): `<comma-separated list>`

**Acceptance criteria:**
- Approved list omitted from prompt if empty.
- Rejected list omitted from prompt if empty.
- Tags in approved list should be reused in preference to inventing new ones.
- Tags in rejected list must not appear in output.
- `suggestTags` uses a simpler prompt (tags-only) but same guidance.

---

## 5. Tag Display Rules

### 5.1 Item list (popup and web app)

- Only **approved** tags are shown as clickable chips on item rows.
- Pending and rejected tags are not shown in the list.

### 5.2 Reader / modal view

- **Approved** tags: displayed with green styling, no action buttons.
- **Pending** tags: displayed with yellow/amber styling, with inline ✓ (approve) and ✗ (reject) buttons.
- **Rejected** tags: hidden (filtered out before rendering).

### 5.3 Tag status map (client-side)

Both the popup and the web app maintain a `tagStatusMap: Record<string, string>` built from `GET /tags` response. This cache governs display and is refreshed after every approve or reject action.

---

## 6. Tag Review UIs

### 6.1 Browser Extension — Pending Tags Bar

- Displayed when `pending.length > 0`; hidden otherwise.
- Shows: "N tag(s) pending review" + "Review →" button.
- "Review →" opens `http://127.0.0.1:3737/` in a new tab (does not provide inline review in the bar itself).

### 6.2 Browser Extension — Modal Inline Review

- When reading an item, pending tags for that item show ✓ / ✗ inline.
- Approving: `POST /tags/approve { tag }` → badge style changes to `approved`, action buttons removed, `tagStatusMap` updated, pending bar count refreshed.
- Rejecting: `POST /tags/reject { tag, itemId }` → badge removed from modal, `tagStatusMap` updated, pending bar count refreshed.

### 6.3 Web App — Tags Panel

- Opened via header "Tags" button (shows "⚑ N pending" when `pending.length > 0`).
- Lists all pending tags with associated item title (truncated, ellipsis).
- Per-row: tag name, item title, ✓ (approve) and ✗ (reject).
- Approval / rejection calls `handleTagAction`, which POSTs and then calls `loadTags()` to refresh all tag state.
- Closed by × button or clicking the backdrop overlay.
- Empty state: "No pending tags — all clear."

### 6.4 Web App — Reader Pane Inline Review

- Same behavior as extension modal: approved tags shown green, pending with buttons, rejected hidden.
- Tag actions refresh `tagData` and `tagStatusMap` via `loadTags()`.

## 7. API Endpoints

### `GET /tags`

**Response:**
```json
{
  "approved": ["tag-a", "tag-b"],
  "pending": [
    { "tag": "tag-c", "itemId": "item-123", "itemTitle": "Article Title" }
  ],
  "rejected": ["tag-d"]
}
```

- `approved`: alphabetical.
- `pending`: ordered by `created_at DESC`. `itemId` and `itemTitle` come from the first `done` item containing that tag. Both may be empty strings if no matching item exists.
- `rejected`: alphabetical.

### `POST /tags/approve`

**Request:** `{ "tag": "<string>" }`
**Response:** `{ "ok": true }`
**Error:** `400 { "error": "Missing tag field" }` if `tag` absent.

### `POST /tags/reject`

**Request:** `{ "tag": "<string>", "itemId"?: "<string>" }`
**Response:** `{ "ok": true }`
**Error:** `400 { "error": "Missing tag field" }` if `tag` absent.

Side effect: if `itemId` is non-empty, re-tagging is triggered asynchronously.

---

## 8. Edge Cases

| Scenario | Behavior |
|---|---|
| Approve a tag that doesn't exist in the DB | `UPDATE` matches zero rows; no error; `GET /tags` continues to work |
| Reject a tag with no itemId | Status set to rejected, no re-tagging |
| Re-tag item with no transcript | `retag` exits early, tags unchanged |
| `suggestTags` returns [] on re-tag | `retag` exits early, tags unchanged |
| Re-tag while Ollama is down | Error silently discarded (`.catch(() => {})`); item's tags remain as-is |
| Same tag suggested for two items simultaneously | `upsertTag` is idempotent; second call is a no-op if already pending |
| Approved tag in LLM output on re-tag | Still registered via `upsertTag('pending')` but conflict rule keeps it `approved` |

---

## 9. Out of Scope

- Bulk approve/reject all pending tags
- Tag rename
- Tag merge
- Tag search or browse page
- Tag usage counts
- Undo approve/reject
