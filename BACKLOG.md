- **Relay architecture: two-tier local+remote** (2026-04-16) — Two design directions to evaluate: (A) **Two-tier**: each machine runs a local relay, sessions connect via Unix socket (fast IPC, OS-managed liveness), local relays federate over network (Tailscale/EC2). Clean separation — local delivery is IPC, cross-machine is network. Evolve transport layer independently. (B) **Single remote relay**: all agents connect directly to one central relay server over the wire (WebSocket/SSE). Every agent always connected to one server. Simpler topology, no federation needed, agents discoverable globally from day one. Downside of (A): federation complexity, two hops for cross-machine messages, local relay is single point of failure per machine. Downside of (B): relay server is global single point of failure, network latency on every local message, EC2 cost + ops burden. **Discuss before deciding.** Owner: chief-of-staff + CEO. Prerequisite: none.

- **Relay redesign: connection-as-registration + store-and-forward** (2026-04-16) — Current relay has two bugs: (1) registration race — agent can be alive and sending but not registered as a recipient (message-relay case); (2) in-memory registry lost on relay restart. Root cause: relay-channel plugin uses HTTP POSTs for sends but has no persistent inbound connection for receiving. Fix: relay-channel plugin maintains a persistent SSE/WebSocket connection to the relay. That connection IS the registration — alive = routable, dropped = immediately unregistered, no stale entries, no TTL management, no files. Add store-and-forward: relay queues messages by agent name, flushes on reconnect. Agents never miss messages even across restarts. Supersedes the file-based registry idea (which has stale-name and scale problems). Owner: message-relay team. Prerequisite: none — this is the right architecture direction.

- **File-based agent registry with TTL** (2026-04-16) — Replace relay's in-memory agent registry with a filesystem directory (`relay/registry/{agent-name}.json`). Each agent's relay-channel MCP plugin touches its file every N seconds with its address. Relay reads the directory for routing. File older than TTL = offline; file absent = never registered. Benefits: survives relay restarts (registry persists on disk), survives registration races (file written on plugin start, no handshake), self-healing (stale entries expire automatically), inspectable (`ls relay/registry/`). Eliminates the current "agent alive but not registered" half-state (e.g. message-relay can send but can't receive). Relay becomes optional for routing — agents remain routable via their file even if relay restarts. Owner: message-relay team. Prerequisite: none.
