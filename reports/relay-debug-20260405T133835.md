# Relay Delivery Debug Report

**Date:** 2026-04-05T13:38:35
**Investigator:** relay-debug agent (spawned by team-lead)

## Root Causes

### 1. CEO has no channel — messages pile up undelivered (PRIMARY)

The relay delivers messages via MCP channel plugins (`POST /deliver` to agent's local port). The CEO does not run a Claude agent session, so there is no channel registration for "ceo". Messages destined for CEO are enqueued, channel delivery is attempted, fails silently (no port registered), and the message stays `delivered: false` forever.

**Impact:** 1,655 undelivered messages in `ceo.jsonl` (736KB), inflating `queue_depth` to 3,342.

### 2. WebSocket broadcast does not mark CEO messages as delivered

The relay broadcasts every message to dashboard WebSocket clients (line 2026-2038 of `index.ts`). The CEO's phone app connects via WebSocket to receive these. However, the broadcast never calls `markDelivered()` for CEO-bound messages, so even when the phone IS connected and receives the message in real-time, the queue file still shows `delivered: false`.

**Impact:** Queue grows unboundedly. The `/status` endpoint reads all queue files synchronously on every call, so queue bloat also degrades API performance.

### 3. No message replay for CEO on WebSocket reconnect

When a dashboard client connects, it receives a state snapshot (agents, backlog, reports, proposals, etc.) but NOT pending CEO messages. If the phone was disconnected when a message arrived, that message is lost to the CEO — never resent on reconnect.

**Impact:** CEO intermittently misses messages when the phone connection drops briefly.

### 4. ffmpeg voice errors (unrelated, pre-existing)

Error logs show repeated ffmpeg failures when processing `.m4a` voice uploads (exit code 183). This is unrelated to delivery but worth noting — voice messages from the phone app fail transcription.

## Fixes Applied

### Fix A: Mark CEO messages delivered on WebSocket broadcast

In `src/index.ts`, the `addMessageListener` callback now checks: if `msg.to === 'ceo'` and at least one WebSocket client has `readyState === OPEN`, call `markDelivered(msg)`.

### Fix B: Flush pending CEO messages on WebSocket connect

When a new dashboard client connects (`/ws` upgrade handler), the server now loads all undelivered CEO messages via `loadUndelivered('ceo')`, sends each to the new client as a `message` event, and marks them delivered. This ensures no messages are lost during phone disconnects.

### Fix C: Export markDelivered from delivery.ts

`markDelivered` was a private function in `delivery.ts`. Exported it so `index.ts` can use it for CEO WebSocket delivery.

### Fix D: Queue cleanup

- Marked 1,657 old undelivered CEO messages as delivered (they were historical, not worth replaying)
- Pruned `ceo.jsonl` from 1,707 lines to 50
- Queue depth dropped from 3,342 to 32
